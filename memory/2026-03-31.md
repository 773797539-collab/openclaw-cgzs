## 07:00-07:15 重大故障记录

### 执行环境完全卡死（04:35-07:00）
- 04:10 后所有 exec 命令全部超时/卡死，持续约 2.5 小时
- 期间 snapshot 仍在刷新（done: 1445→1541），说明后台 task_executor 在跑
- 但我的 tool 执行端完全无响应

### task_executor 多实例乱战
- 05:10 恢复后发现：4个进程在抢同一份 state 文件
- inbox-disp state 文件（/tmp/inbox-disp-state.json）被并发写坏（coreIdx 跳到 2/3）
- 证据：journalctl 多次显示同一批任务被重复派发

### 修复定位：dispatch_one idle 时返回值的 bug
- 原 bug：inbox-disp 返回 idle 时，dispatch_one() 打印日志但返回 None
- 外层：`if idle_reason is None` → 进入了 sleep(CORE_IDLE_SLEEP) 分支
- **但真正的问题**：inbox-cron.sh 每 30 秒启动一次，task_executor 每次从头跑 while 循环
- task_executor 跑完一批白名单任务（30秒内）后进入 idle，然后 sleep(CORE_IDLE_SLEEP=180)
- **但 inbox-cron 每 30 秒就重启一个新的 task_executor 进程**，打断了正在 sleep 的旧进程
- 新进程看到 doing=空，又立即 dispatch，覆盖了旧进程的 sleep 等待

### inbox-cron.sh vs task_executor.py 角色混乱
- inbox-cron.sh：原本设计为"每 30 秒扫描一次 inbox"，通过 inbox-disp 补货+派发
- task_executor.py：原本设计为"持续执行 doing 中的任务"
- 但 inbox-disp.js 同时被两者调用：inbox-cron.sh 调它补货，task_executor 调它派发
- 两者共用 state 文件，导致派号偏移

### 关键修复方向
1. task_executor 的 idle sleep 不能被 inbox-cron 的 30 秒轮询打断
2. 两个 daemon 不能共用同一个 state 文件
3. 或者：task_executor 内部 self-dispatch，inbox-cron 只做备份
