## P-003：Token 字段语义误读（最严重）
**日期**: 2026-03-27
**级别**: P1

**问题描述**：
两次将 `current_interval_usage_count` 误读为"已用"：
- 第一次读 4216，误判为"已用 4216/4500"
- 第二次读 4205，误判为"已用 93.4%，已触发停止线"

**根因**：
`current_interval_usage_count` 字段名含 `usage`，直觉判断为"已使用"，但实际语义是**剩余额度**（remaining）。当前实测映射与字段名不一致，项目内暂按 remaining 处理，待官方文档进一步确认。

**验证（以官方后台为准）**：
```
22:17 官方后台显示：
  已用：308 / 4500（约 6.8%）
  剩余：4192 / 4500（约 93.2%）

API 原始字段值：
  current_interval_usage_count = 4192 ← 实际是"剩余"，不是"已用"
  total_count = 4500
  推算已用 = 4500 - 4192 = 308 ✅（与官方吻合）

数值变化规律（验证语义）：
  4216 → 4205 → 4192（逐步下降）= 剩余额度在消耗
  如果是"已用"，应当逐步上升，但它在下降 → 只能是剩余额度
```

**正确理解**：
  - 当前实测映射与字段名不一致，项目内暂按 remaining 处理，待官方文档确认
- 已用 = `total_count - usage_count`
- 停止线：剩余 ≤ 20% 时停止（即 usage_count ≤ 900）

**本轮实际状态**：
- 剩余 4192/4500（93.2%），已用 308/4500（6.8%）
- 停止线未触发，本周期可继续工作

**教训**：
1. 字段名含 `usage` 不一定代表"已用"，要结合数值变化规律判断
2. 数值从 4216→4205→4192 逐步下降，这更符合"剩余额度被消耗"而非"已用增加"
3. 遇到命名与直觉不符的字段，必须做反向校验（变化方向、数值合理性）
4. MiniMax Token Plan：以后以官方后台为准，API 值作为交叉验证

## P-004：get_zt_pool 语义错误
**日期**: 2026-03-27
**级别**: P2

**问题**：`get_zt_pool()` 注释写"获取昨日涨停股池"，但 push2 f3 字段返回的是**今日**涨幅数据。

**现象**：热景生物 f3=20%，代表今日涨停，不是昨日涨停。

**影响**：回调股筛选逻辑实际上是在"今日涨停股中找回调"，而非"昨日涨停今日回调"。在强势市场中两者结果相似（都无回调），但在弱势市场中会完全不同。

**建议**：区分两个概念——"昨日涨停股池"（需要前一日数据）和"今日涨停股池"（当前 push2 返回）。选股系统应以前者为目标，但目前数据源不支持前一日，只能用后者近似。
