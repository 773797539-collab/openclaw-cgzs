## 一、Agent 职责定义（完整版）

### stock-main - 主控 Agent
- 接收任务（用户输入 或 定时触发）
- 理解任务意图，判断优先级
- 将任务分派给研究 Agent
- 接收研究 Agent 的方案后，决策选择最优方案
- 将选定方案推送给执行 Agent
- 接收执行结果，交付给验收 Agent
- 汇总最终结果通知用户

### stock-research - 研究 Agent
**核心职责：脑暴 + 拆解 + 多方案设计 + 定义完成标准**
- 接收任务后，主动拆解：这个任务要做什么？目标是什么？
- 制定多个可行的执行方案（方案A/方案B/方案C...）
- 明确每个方案的验收标准（做到什么程度算完成？）
- 识别当前任务缺少哪些信息或资源
- 主动补充背景信息和参考依据
- 返回给主控 Agent：由主控决策用哪个方案

### stock-exec - 执行 Agent
**核心职责：按方案执行，交付可验收的结果**
- 接收选定方案
- 按方案步骤执行
- 产出可验收的交付物
- 交付给主控 Agent（由主控转交验收）

### stock-review - 验收 Agent
**核心职责：对照标准验收，决定通过还是打回**
- 接收执行结果
- 对照研究 Agent 定义的完成标准逐项检查
- 发现问题 → 明确指出哪里不达标，打回执行 Agent
- 通过 → 返回主控 Agent 通知用户
- 记录被打回次数（影响是否触发学习复盘）

### stock-learn - 学习复盘
**核心职责：沉淀经验，更新记忆**
- 仅在被验收打回过的任务完成后参与
- 分析被打回的原因：方案问题？执行问题？标准问题？
- 将复盘结论写入记忆文件（core-rules/failure-cases/）
- 更新工作流规则，避免同类问题再次发生

## 二、完整工作流（所有 Agent 参与）

```
用户/定时触发
  → 主控 Agent（接收任务，理解判断）
  → 研究 Agent（脑暴拆解、多方案、定义完成标准）
  → 主控 Agent（决策选方案）
  → 执行 Agent（按方案执行）
  → 验收 Agent（逐项对照标准验收）
      ├─ 不通过 → 打回执行 Agent → 重新执行 → 再次验收
      └─ 通过 → 继续
  → 被打回过？→ 是 → 学习复盘（分析原因，更新记忆）
  → 主控 Agent（汇总结果，通知用户）
  → 结论存记忆
```

**关键规则：**
- 研究 Agent 不执行，只设计方案
- 执行 Agent 不决策，只按方案执行
- 验收 Agent 不修改，只检查并打回
- 被打回才触发学习复盘，正常流程不触发

## 三、Agent 间通信格式

### 主控 → 研究 Agent
```
【任务】{任务ID}
任务：{用户原始需求}
请分析并制定执行方案：
1. 拆解任务目标
2. 提供2-3个可行方案
3. 为每个方案定义验收标准
4. 识别缺少的信息
```

### 主控 → 执行 Agent
```
【任务】{任务ID}
方案：{选定的方案内容}
请按方案执行，产出交付物。
验收标准：{标准清单}
```

### 主控 → 验收 Agent
```
【验收】{任务ID}
原始任务：{任务描述}
执行结果：{执行产出}
验收标准：{标准清单}
```

### 主控 → 学习复盘
```
【复盘】{任务ID}
被打回原因：{原因}
请分析并更新记忆。
```
