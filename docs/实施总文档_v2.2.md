# OpenClaw 项目实施总文档 V2.2

**可交付执行版 | 面向从零到一实施 | 含多项目扩展、留痕治理、主文档持续维护规则**  
**日期：2026-03-26**

## 一、文档定位与使用方式

这是一份可以直接交给 OpenClaw 执行的主实施文档。目标不是泛泛介绍，而是让执行者基于本文，完成从单项目到多项目的长期在线 AI 系统搭建，并且满足以下硬要求：

- 所有关键改动必须留档，可复盘，可回滚。
- 每次更新必须有更新记录，能看出改了什么、为什么改。
- 项目主文档必须持续维护，不允许“系统已经变了，文档还停留在旧版本”。
- 当前第一业务场景是 A 股辅助，但整体结构必须预留未来扩展到其他项目的能力。
- 系统面向单一可信操作者，不做多租户产品设计。

本文默认适用于以下场景：

- 部署在一台独立云电脑 / 云服务器上。
- 用户计划使用 OpenClaw 最新稳定版本。
- 模型主要走云端 API，而不是在本机做大模型推理。
- 允许较高权限，但仍要求做好日志、备份、变更记录和回滚。

## 二、已知前提、事实基线与推定范围

### 2.1 已知前提

- 本次实施将以 OpenClaw 最新稳定版本为基线。
- 当前还没有正式创建这套系统的各项目 agent 与实际目录。
- 当前部署环境是与本地隔离的云电脑 / 云服务器。
- 用户愿意给 OpenClaw 较高权限，但希望所有关键操作可追溯、可复盘、可回滚。

### 2.2 版本与官方能力基线

按当前官方资料，OpenClaw 稳定版已更新到 2026.3.23。  
同一 Gateway 下可以创建多个隔离 agent；每个 agent 拥有独立 workspace、独立 agentDir、独立 session store。  
大多数配置变更支持热重载；Gateway 服务端口等关键变更仍可能触发重启。  
工作区 plugin 不再允许“打开仓库即隐式自动加载”，显式信任已成为安全基线。  
plugin 属于进程内扩展，不是沙箱执行，应按高风险可执行扩展看待。

### 2.3 推定范围（后续落地时再按实机修正）

以下内容属于基于常见 VPS / 云电脑场景的工程推定，不是官方硬指标：

- 多项目容量规划建议
- 不同项目数量对应的 CPU / 内存建议
- 多项目并发时的调度与限流建议
- 项目目录命名和推荐的 agent 拆分数量

如本地资源、模型供应商限额、通知渠道、浏览器自动化强度与本文推定不一致，执行时应先记录差异，再走最小影响修正，不得直接推翻主结构。

## 三、项目目标、边界与成功标准

### 3.1 项目目标

1. 把 OpenClaw 从“会聊天的机器人”升级成“长期在线、可调度、可留痕、可复盘、可扩展”的个人 AI 指挥系统。
2. 当前第一核心业务是 A 股辅助，但系统架构不能只写死股票；要能复制到其他业务项目。
3. 门户站要成为主可观测中心，避免把关键状态全部淹没在聊天记录里。
4. 所有面向用户的输出默认中文、结论优先、少打扰。
5. 系统应当具备成长能力，但成长任务不能压过主业务任务。

### 3.2 明确非目标

- 不自动下单，不直连券商资金账户。
- 不把控制面直接裸露在公网。
- 不在没有备份和回滚方案的情况下做高风险系统改动。
- 不把测试脚本、临时修补、重复日志当正式交付。
- 不把“24 小时运行”理解为无脑空转。

### 3.3 成功标准

系统达到以下状态，视为项目一期成功：

- 能稳定接收用户消息、定时任务与内部成长任务。
- 能按项目拆分 agent、记忆、任务、日志和结果。
- 能在门户页看到关键状态、当前任务、通知摘要、异常、备份健康。
- 能对每次关键改动留下变更记录、执行记录和更新日志。
- 能把主文档持续维护下去，而不是只在初次交付时写一次。

## 四、总体架构设计

系统采用“共享基础设施 + 项目级隔离脑 + 统一治理层”的结构。

### 4.1 共享基础设施层

这部分可以被多个项目共用：

- OpenClaw Gateway
- 基础模型配置
- Portal 外壳与统一前端框架
- Tunnel / Access 接入层
- 备份与校验机制
- 通知通道底层
- 通用日志归档、健康检查、告警框架

### 4.2 项目隔离层

每个项目必须拥有自己的：

- project_id
- 独立 workspace
- 独立主 agent
- 独立项目记忆
- 独立任务池、结果池、日志目录、trace 目录
- 独立项目级文档与变更日志

### 4.3 治理层

治理层跨项目统一，但必须能区分项目：

- 变更审批规则
- 任务优先级与并发配额
- 备份保留策略
- 文档主版本管理
- 更新日志规范
- 回滚机制
- 门户统一概览页

## 五、多项目扩展总原则

### 5.1 同一台云电脑能否部署多个项目

可以。  
推荐方式不是“一台机器只能有一个项目”，而是“同一可信操作者边界下，一台机器可以跑一个 Gateway + 多个项目 agent / workspace”。

### 5.2 多项目能否同时运行

可以，但要区分“技术上能同时跑”和“业务上建议如何并发”：

- 技术上：多个项目 agent 可以同时存在、同时收消息、同时执行心跳 / cron / 研究任务。
- 实际上：是否顺畅，取决于 CPU、内存、浏览器自动化数量、模型供应商并发额度、网络质量、日志量与任务峰值。

### 5.3 多项目是否必须一项目一整套 Gateway

默认不需要。  
对于你这种“同一操作者、同一台云电脑、同一信任边界”的场景，推荐：

- 一个 Gateway
- 多个项目 workspace
- 多个项目主 agent
- 共享基础设施
- 独立项目记忆与项目级任务系统

只有在以下场景，才建议拆分成多个 Gateway，甚至多个 OS 用户或多个主机：

- 项目之间信任边界完全不同
- 某项目接入高风险插件或高危脚本
- 某项目必须使用独立浏览器身份、独立凭证、独立网络出口
- 某项目压力太大，已经明显影响其他项目稳定性

## 六、Agent 设计与隔离策略

### 6.1 默认推荐的项目级 agent 组

每个项目默认至少配置以下 5 类 agent：

1. 前台总控 Agent  
   负责理解用户需求、分派任务、汇总结果、决定优先级。

2. 研究 Agent  
   负责资料检索、规则整理、方案草拟、候选信息归纳。

3. 执行 Agent  
   负责脚本执行、文件改动、配置调整、数据产出。

4. 验收 Agent  
   负责检查结果是否达到预期、是否满足门禁、是否能交付。

5. 学习与复盘 Agent  
   负责复盘失败样本、沉淀经验、优化提示词与工作流。

### 6.2 哪些 agent 必须按项目新建

以下角色必须项目独立，不得共用长期记忆：

- 前台总控 Agent
- 研究 Agent
- 执行 Agent
- 验收 Agent
- 学习与复盘 Agent
- 项目 MEMORY / memory / bank / docs

原因：这些角色都会直接读取项目规则和项目上下文，若共用，极易产生跨项目记忆串扰。

### 6.3 哪些能力可以共享

以下能力可以共享底层，不必每个项目重复搭：

- Gateway
- Portal 外壳
- Tunnel
- 备份服务
- 健康检查服务
- 通知通道底层
- 通用监控与日志归档框架

### 6.4 前端播报与用户沟通 agent 是否必须新建

结论：默认不建议一开始就共享“会思考”的沟通 agent。  
更稳的做法是：

- 共享“无记忆、无决策、只负责格式化与转发”的通知适配层。
- 项目级真正负责理解业务和生成结论的，仍由项目主 agent 负责。
- 如果后期一定要共享一个“用户沟通中心 agent”，必须限制它不直接读取多个项目的长期记忆，只能消费标准化项目摘要。

### 6.5 Agent 隔离判定表

| 组件 | 是否项目独立 | 默认建议 |
|---|---|---|
| 前台总控 Agent | 是 | 必须独立 |
| 研究 Agent | 是 | 必须独立 |
| 执行 Agent | 是 | 必须独立 |
| 验收 Agent | 是 | 必须独立 |
| 学习与复盘 Agent | 是 | 必须独立 |
| 项目记忆 | 是 | 必须独立 |
| Gateway | 否 | 共享 |
| Portal 外壳 | 否 | 共享 |
| Tunnel / Access | 否 | 共享 |
| 备份与健康检查 | 否 | 共享 |
| 通知底层 | 否 | 共享 |
| 通知格式化层 | 视实现而定 | 初期共享、禁止项目记忆 |

## 七、权限、安全与信任边界

### 7.1 当前场景的信任模型

当前项目按“单一可信操作者边界”设计。  
这意味着：

- 可以在同一 Gateway 下运行多个项目 agent。
- 可以给系统较高权限，但要把“能做”和“应该做”分开。
- 高权限不等于所有 agent 全都拥有同样权限。

### 7.2 权限策略

即使当前准备给 OpenClaw 较高权限，也应采用分层策略：

- 前台总控 Agent：优先读权限、调度权限、少量写权限。
- 研究 Agent：优先网络 / 读取 / 总结权限。
- 执行 Agent：需要更高文件与脚本执行权限，但必须配套留痕和回滚。
- 验收 Agent：只读优先，避免边验边改。
- 学习与复盘 Agent：读取项目结果、日志、失败样本，不直接做高危改动。

### 7.3 关于 plugin、hook、workspace 扩展的安全要求

- plugin 视为高风险可执行扩展。
- hook 虽然更轻，但同样属于会影响行为的本地代码。
- 工作区内发现的扩展，不得默认视为“安全、自动生效、可放心启用”。
- 任何第三方扩展都必须先记录来源、用途、风险、启用原因和回滚方法。

### 7.4 强制安全底线

- 不把主控制面开放给不受控公网。
- 不把敏感凭证写进对外可见结果文件。
- 不让高风险脚本在无 trace、无日志、无回滚点的情况下直接运行。
- 不在同一项目中同时启用多个功能重叠、来源不明的扩展。

## 八、留痕、日志、更新记录与回滚规范

这是本项目必须强化的一章，不能弱化成“尽量记录”。

### 8.1 每次关键操作必须产出三类记录

1. 变更记录  
   记录改了什么、为什么改、影响哪些模块、如何回滚。

2. 执行记录  
   记录执行了哪些命令、关键输出是什么、是否失败重试。

3. 文档更新记录  
   如果改动影响架构、流程、目录、规则、接口或运行方式，必须同步更新主文档。

### 8.2 推荐记录目录

- `docs/changelog/`：项目更新记录
- `runtime/traces/`：执行 trace
- `logs/`：系统运行日志
- `reports/`：阶段性结果、验收、复盘
- `docs/decisions/`：重要设计决策记录
- `docs/failure-cases/`：失败样本与踩坑记录

### 8.3 变更记录最少字段

每一条变更记录至少包含：

- 时间
- project_id
- 操作者 / agent
- 改动目标
- 改动文件
- 风险等级
- 验收点
- 结果
- 回滚方式
- 是否已同步主文档

### 8.4 更新日志最少字段

每次版本或结构更新，至少记录：

- 更新时间
- 当前 OpenClaw 版本
- 本次新增
- 本次修改
- 本次删除 / 停用
- 是否影响既有项目
- 是否需要用户人工确认
- 是否需要重新备份 / 校验

### 8.5 回滚原则

- 所有高风险改动前必须先创建可定位的回滚点。
- 能旁路验证的，先旁路，不直接切正式链路。
- 能单模块回滚的，不做整项目回滚。
- 高风险更新默认先备份，再变更，再校验。

## 九、主文档持续维护规则

### 9.1 文档主版本规则

为保证 OpenClaw 后续能持续自动维护文档，本项目不建议把 DOCX 当作唯一权威源。  
推荐规则如下：

- `实施总文档.md`：唯一主文档，供 OpenClaw 持续自动更新。
- `实施总文档.docx`：阅读友好版，按阶段或关键变更后导出覆盖。
- 桌面副本：仅作为查看入口，不作为唯一权威源。

### 9.2 推荐保存位置

- 主文档：`<PROJECT_ROOT>/docs/实施总文档.md`
- 阅读版：`<PROJECT_ROOT>/docs/exports/实施总文档.docx`
- 桌面镜像：`~/Desktop/<project_id>_实施总文档_最新.docx`
- 更新日志：`<PROJECT_ROOT>/docs/changelog/project-change-log.md`

### 9.3 对 OpenClaw 的明确要求

在项目初始化阶段，OpenClaw 必须完成以下动作：

1. 在项目 workspace 内创建 `docs/` 目录。
2. 保存一份 Markdown 主文档作为后续持续维护源。
3. 导出一份 DOCX 阅读版。
4. 在桌面放一份最新阅读版镜像，方便人工查看。
5. 任何涉及架构、目录、流程、接口、agent 拆分、权限策略变更时，先更新 Markdown 主文档，再覆盖导出 DOCX。

### 9.4 不允许的做法

- 只改系统，不改文档。
- 只在聊天里说明，不落正式文档。
- 只改桌面副本，不改项目内主文档。
- 多份文档同时手改，导致版本漂移。

- 任务、结果、通知、异常、资产等数据分开存放。
- 测试数据与正式数据分离。
- 临时文件可归档、可清理、可追溯来源。

## 十一、记忆与知识沉淀规范

### 11.1 记忆分层

建议采用四层：

1. 核心规则层  
   项目目标、边界、口径、禁区、验收硬规则。

2. 流程与模板层  
   SOP、命令模板、结果模板、复盘模板、通知模板。

3. 经验样本层  
   失败案例、成功案例、调优记录、例外处理。

4. 运行态层  
   短期状态、当前上下文、临时任务说明。

### 11.2 记忆隔离原则

- 不同项目的核心规则层不得共用。
- 不同项目的经验样本层默认不得混写。
- 共享的只能是“通用基础模板”与“平台级组件经验”，不能是项目业务判断。
- 前台总控 agent 不得读取不相关项目的长期记忆。

### 11.3 记忆更新规则

- 新规则进入长期记忆前，必须有来源。
- 失败样本必须写结论，不允许只留一堆错误日志。
- 已过时规则要标注失效日期或替代方案。
- 每次大版本更新后，至少复核一次核心规则层。

## 十二、任务系统、审批与阻塞机制

### 12.1 任务分类

- 主业务任务
- 运维任务
- 成长任务
- 风险任务
- 文档任务

### 12.2 优先级规则

默认优先级：

1. 用户明确下发的主业务任务
2. 当前阻塞主链路的问题
3. 高风险故障修复
4. 日常运维
5. 成长与优化任务

### 12.3 审批要求

以下任务建议先进入审批池：

- 高权限脚本执行
- 插件启停
- 目录大迁移
- 删除历史数据
- 会影响多个项目的共享层改动
- 有概率中断服务的更新动作

### 12.4 阻塞机制

阻塞任务必须单独入池，并写清楚：

- 阻塞点
- 影响范围
- 当前规避方案
- 所需人工输入
- 下次重试条件

### 12.5 文档任务不可忽略

凡是影响架构或长期流程的改动，都要自动生成一个“文档同步任务”，直到主文档更新完成才能关闭。

## 十三、通知、少打扰与门户标准

### 13.1 通知原则

- 少而准，不刷屏。
- 用户默认先看结论，再决定是否展开细节。
- 可以汇总的，不拆成多条。
- 能放门户页展示的，不全部挤到聊天里。

### 13.2 必须主动通知的情况

- 主业务结果产出
- 关键失败
- 高风险动作待审批
- 备份失败
- Gateway / 渠道异常
- 文档与系统状态不一致
- 需要用户人工处理的真实阻塞

### 13.3 门户必须具备的模块

- 状态总览
- 当前任务与任务池
- 审批池
- 阻塞池
- 最近结果
- 最近异常
- 备份健康
- 记忆健康
- 更新日志摘要
- 多项目概览页

### 13.4 中文输出要求

- 所有面向用户的默认输出使用中文。
- 接口字段可以保留英文 key，但展示层必须中文化。
- 状态页标题、按钮、摘要、告警默认中文。

## 十四、多项目并发与容量规划（推定）

本章为工程推定，后续应根据你实际模型、任务密度、浏览器自动化强度、通知频率再校正。

### 14.1 判断资源压力时看什么

资源压力不只取决于项目数量，更取决于：

- 是否频繁跑浏览器自动化
- 是否频繁执行 shell / Python / node 脚本
- 是否开启大量心跳、cron、hook
- 是否保留大量日志与 trace
- 模型接口是否有并发限流
- 项目是否经常同时高峰运行

### 14.2 粗略容量建议

#### 1 个项目
- 最低建议：2-4 vCPU / 4-8 GB RAM
- 更稳建议：4 vCPU / 8-16 GB RAM
- 适用：轻量单项目、主要走云端 API、少量自动化

#### 2 个项目
- 最低建议：4 vCPU / 8-16 GB RAM
- 更稳建议：8 vCPU / 16 GB RAM
- 适用：两个项目都在线，但浏览器自动化不重

#### 5 个项目
- 最低建议：8 vCPU / 16 GB RAM
- 更稳建议：8-16 vCPU / 32 GB RAM
- 适用：多项目常驻、定时任务较多、门户和日志都保留

#### 10 个项目
- 最低建议：16 vCPU / 32 GB RAM
- 更稳建议：16-32 vCPU / 64 GB RAM
- 适用：多个项目长期同时运行，且存在较高并发、较多自动化或较重爬取任务

### 14.3 结论

- 同一台机器可以跑多个项目，不是“一机一项目”。
- 但项目越多，越不能靠“所有任务都同时放开跑”的方式。
- 应当引入项目级并发配额和高风险任务串行规则。

### 14.4 推荐并发规则

- 高风险执行任务：全局串行
- 一般研究任务：可多项目并发，但需设上限
- 门户刷新、摘要生成、轻量通知：可并发
- 浏览器自动化：按资源单独限流
- 同一项目内执行链：优先保证先后顺序，避免结果互相覆盖

### 14.5 资源不够时的优先降级顺序

1. 降低成长任务频率
2. 降低门户非关键模块刷新频率
3. 合并低优先级通知
4. 限流研究型批处理
5. 暂停重型浏览器自动化
6. 最后才动主业务任务

## 十六、多 Agent 工作流操作（实测版）

### 16.1 工作流状态
- **main（主控）**：协调、派发、监控、汇总
- **stock-research**：研究分析（`sessions_spawn` 派发，`runtime=subagent`）
- **stock-exec**：执行修正（审查发现问题后派发）
- **stock-review**：验收评审（对研究结果质量把关）

### 16.2 派发标准流程
```
用户需求 → main 理解 → 派发 stock-research → 产 出报告
→ 派发 stock-review → 评分+问题清单
→ 若有问题 → 派发 stock-exec 修正
→ 完成 → 更新 tasks.json + 门户页
```

### 16.3 派发命令示例
```python
sessions_spawn(
    task="任务描述",
    label="任务标签",
    runtime="subagent",
    agentId="stock-research",  # 可选，不填则用默认agent
    runTimeoutSeconds=120,
    mode="run"
)
```

### 16.4 关键规则
- 每次派发后必须更新 `portal/status/tasks.json`（doing/todo/done）
- 每个 subagent 完成后必须派发 stock-review 验收
- 审查发现问题必须派发 stock-exec 修正，不直接覆盖
- 修正完成后手动合并结果并更新 tasks.json
- 所有工作流状态实时反映在 `portal/index.html`（运行 build_index.py）

## 十八、交付给 OpenClaw 的明确执行准则

1. 先搭治理底座，再搭业务模块，不得反过来。
2. 任何关键改动前先建回滚点。
3. 任何影响结构的改动后先更新主文档，再继续后续动作。
4. 任何新项目默认新建独立 project_id 与项目级主 agent。
5. 共享基础设施可以复用，但项目业务记忆不得混用。
6. 没有执行记录、变更记录、更新日志的改动，不算完成。
7. 没有验证通过的模块，不得宣称“已完成”。
8. 遇到本地现状与本文冲突时，不得直接推翻设计，必须先记录差异与修正理由。
