## 踩坑 1：portal.js 写入不完整（工具截断）

**发生时间**：2026-03-27 03:xx
**严重程度**：P0
**影响**：portal.js 语法错误，页面 Loading 状态

**报错信息**：
```
node --check portal/static/portal.js
SyntaxError: Unexpected end of input (line 433)
```

**根因**：
用 heredoc（bash `<<EOF`）写 JS 文件时，Python 代码中的 `$` 符号被 bash 解释，且 write/exec 工具有 ~170 行截断限制，多行 heredoc 无法完整写入。

**尝试过的失败方案**：
1. `exec` 工具 heredoc - `$()` 和 `'` 造成引号转义爆炸
2. `write` 工具 - 超过170行自动截断
3. `python3 - <<EOF` - 同样有 heredoc 引号问题
4. 分两次 write - 写到一半就在同一位置截断

**最终解决方案**：
用 `write` 工具写一个生成脚本（<170行），通过 `exec` 执行该脚本，每个脚本追加写入固定行数到 JS 文件。分3个阶段完成：
```
gen2.py (gen3_fixed.py) → gen4.py
```

**教训**：长文件不能用 heredoc 或 write 截断方式写入，必须用自举脚本 + 分段追加。

## 踩坑 2：fetch_live_price 返回 None

**发生时间**：2026-03-27 04:xx
**严重程度**：P0
**影响**：持仓价格显示 0

**报错信息**：
```
Portfolio holdings: 1
  立达信 100股 成本¥20.655 × 100 = ¥2065.50
  现价¥None × 100 = ¥0.00  ← fetch_live_price 返回 None
```

**根因**：
`all_status()` 直接读取 `portfolio.json` 中的静态 holdings 数据，没有调用 `fetch_live_price()` 异步函数。`portfolio.json` 里只有 `cost` 和 `shares`，没有 `price`。

而 `/api/status/portfolio` 端点正确调用了 `fetch_live_price()`，所以那个接口能拿到价格。

**解决**：
在 `all_status()` 中，对每个 holding 调用 `fetch_live_price()` 并写入 holdings 数组：
```python
for h in holdings:
    code = h.get("code", "")
    if code:
        live = fetch_live_price(code)
        if live:
            h["price"] = live["price"]
            h["change_pct"] = round(live["pct"], 2)
```

**教训**：`all_status()` 是一个聚合端点，如果某子模块有独立的实时获取逻辑，必须在聚合时也调用它，不能假设数据已经被前置处理过。

## 踩坑 3：HTML id 与 JS id 不匹配

**发生时间**：2026-03-27 03:xx
**严重程度**：P1
**影响**：持仓列表、Agent架构面板无法渲染

**具体情况**：
| 位置 | HTML id | JS 渲染目标 |
|------|---------|-------------|
| 持仓列表 | `holdings-list` | `holding-list` |
| Agent 架构 | `agent-arch` | `agent-list` |
| Agent计数 | `arch-count` | `agent-count` |

**根因**：HTML 和 JS 是分别写的，没有对照检查 id 命名。

**解决**：
```bash
sed -i 's/id="holdings-list"/id="holding-list"/g' portal.html
sed -i 's/id="agent-arch"/id="agent-list"/g' portal.html
```

**教训**：每次写完 JS 函数后，必须对照检查 `document.getElementById()` 的 id 和 HTML 中的 `id="xxx"` 是否完全一致。

## 踩坑 6：subprocess curl 对 MCP HTTP 返回 HTTP 501

**发生时间**：2026-03-28 02:xx
**严重程度**：P0
**影响**：morning_briefing 完全失败

**报错信息**：
```
urllib.error.HTTPError: HTTP 501
# 或
subprocess.run(['curl', '-s', 'http://82.156.17.205:...']) → HTTP 501
```

**根因**：
`subprocess.run(['curl', ...])` 和 `urllib.request.Request(url, method='POST')` 对 MCP 服务器的 HTTP 协议处理不一致，导致返回 501 或数据截断。

**解决**：
统一使用 `urllib.request.Request(url)`，不指定 `method` 参数，让库自动选择：
```python
req = urllib.request.Request(MCP_URL, data=json.dumps(payload).encode(),
    headers={"Content-Type": "application/json",
             "Accept": "application/json, text/event-stream"})
with urllib.request.urlopen(req, timeout=8) as resp:
```

**教训**：MCP HTTP 接口对协议细节敏感，urllib 比 subprocess curl 更可靠。

## 踩坑 7：SSE JSON 提取失败（正则匹配失败）

**发生时间**：2026-03-28 02:xx
**严重程度**：P0
**影响**：morning_briefing 价格解析失败

**SSE 响应格式**：
```
event: message
data: {"jsonrpc":"2.0","id":1,"result":{"content":[...]}}
```

**失败的方案**：
```python
re.search(r'data:\s*({.+?})\s*$', raw, re.DOTALL)  # 正则无法匹配多行 SSE
```

**解决**：
```python
json_start = raw.find('data:')
if json_start >= 0:
    result = json.loads(raw[json_start+5:].strip())
```

**教训**：SSE 是流式格式，`data:` 后面不一定是行尾，不要用行尾锚点 `$` 的正则匹配。

## 踩坑 8：价格行解析错误（含非价格"当日:"行）

**发生时间**：2026-03-28 02:xx
**严重程度**：P1
**影响**：morning_briefing 显示错误价格

**问题**：
`brief` 返回多行包含"当日:"的文本，其中含非价格行（如"6.91"、"1.25"等涨幅/换手率数值），原代码取第一个匹配行导致价格错误。

**解决**：
价格行必须同时含 "当日:" 和 "最高:" 关键字：
```python
price_line = [ln for ln in lines if "当日:" in ln and "最高:" in ln][0]
price = float(price_line.split(":", 1)[1].strip().split()[0])
```

**教训**：解析 MCP 返回的固定格式文本时，不能只匹配单一关键字，要找唯一性特征。
